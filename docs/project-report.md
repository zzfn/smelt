# smelt 项目简述

---

## 一句话

**smelt 是 Mac 上的「AI coding 驾驶舱」——一个专为「同时指挥多个 CLI coding agent 干活」设计的桌面工作台。**

---

## 一、要解决的问题

当 agent 能独立跑完「读代码 → 改代码 → 跑测试 → 提交」整条链路，人的角色就从**打字的人**变成**看导航、下指令的人**。一个人同时开三五个 agent 干活，已经是现实工作方式。

但工具还停在「一个人对着一个终端」的年代。实际遇到的问题：

**1. 不知道该管哪个**
同时跑 5 个 agent，其中一个卡在权限确认上等了半小时，你不知道——只能挨个切窗口去看。agent 的状态（在跑 / 等你批准 / 等你输入 / 干完了）在终端里没有任何结构化表达。

**2. 会话本身太脆——它会死**
关掉终端、GUI 崩溃、装个新版本，正在跑的 agent 全死，上下文没了得从头再来。agent 任务动辄跑十几分钟，这个代价很实。

**3. 人和会话绑死在同一台电脑前——它活着，但你够不着**
会话好好跑着，你人不在这台电脑前就碰不到它。出门、开会、通勤，agent 卡在一句「要不要执行这条命令」上，一等几小时；回到座位才发现整个下午白跑了。

**4. 多项目切换成本高**
每个项目一个窗口、一套布局、一堆 cd 和启动命令，重开一次全要重来。

**5. 一件事要开三个软件**
盯 agent 在终端，看改了什么切 git 客户端，翻代码找文件再开编辑器。三个窗口来回跳，而且换个项目就要挨个重新指一遍路。

**6. 团队之间靠人肉搬运**
以 review 为例：agent 改完一版，同事看出问题贴进飞书，开车的人再手动复制回终端喂给 agent。评论一离开 diff 就丢了上下文，来回搬还容易漏。

### 现有工具为什么不解决

| 方案 | 会话持久化 | agent 状态感知 | 备注 |
|---|---|---|---|
| iTerm2 / Warp | ✗ | ✗ | 通用终端，不认 agent |
| tmux | ✓ | ✗ | 有持久化，但不知道 agent 在等你还是在跑 |
| VS Code / Zed 插件 | ✗ | — | 战场在编辑器，而 agent 干活在终端 |

spec 驱动、看板派活这类前置方法，解决的是「开工前把事情想清楚」，和上面这些痛点是两件事：agent 的失败大多发生在**执行中**且不可预见——权限门（`rm -rf` / `git push` 这类是安全边界，不是信息缺失，规格写得再细也不能替人点「同意」）、规格没覆盖的歧义分支、依赖装不上测试挂了。这些都发生在终端里，且发生时 agent 会停下来等人。而且前置做得越好、能同时跑的 agent 越多，「现在到底哪个在等我」反而越是瓶颈。

## 二、已经做的

### 地基：`smeltd`（终端持久化守护）

**终端会话能脱离人存活。** 普通终端一关 GUI 会话就死，于是压根**不存在**「一个能被围观、能被交接、能被远程遥控的对象」。smeltd 让这个对象存在了——它是后面所有能力的前提，不是一个可选功能。

### 状态感知：走终端协议，不绑死任何一家

靠终端标题（OSC 0/2）+ OSC 9/777 通知 + 响铃识别 agent 状态，是终端通用约定而非某家私有格式，任何遵守约定的 agent 都能被识别（Claude Code 开箱即用）。

会话状态四档着色（等你批准 / 等你输入 / 跑着呢 / 有结果可看），Dock 与菜单栏角标，系统通知（app 在前台自动噤声）。

### 工作台

多项目 × 多标签内嵌真终端（`claude` / `codex` / `vim` / `htop` 等 TUI 都真跑）· 分屏 · 命令面板 · 快捷启动 · 布局自动存档恢复。

### 远程访问

手机浏览器看本机 agent 实时终端画面、批准 / 拒绝权限、点选菜单；装 cloudflared 后一条公网链接出门可用。

### 读写代码

文件树 + 文件名/内容搜索 · Git diff（字符级行内高亮）· 代码热力图（从 git log 提炼改动热点）。

## 三、等待做的

### 1. 任务系统：统一的执行入口

指挥的单位应该是**任务**，不是会话。会话是执行现场，任务才是「要干什么」——人下完指令就该能走开，而不是守着终端一条条喂。

关键在于：**任务从哪来可以有很多种，但执行、状态回报、验收应该只有一套。** 三种来源汇进同一个队列：

| 来源 | 谁创建 | 典型场景 | 现状 |
|---|---|---|---|
| **人工创建** | 人 | 手动派活、定时开跑 | **已上线** |
| **agent 动态创建** | agent 调 smelt 提供的 CLI | 测试挂了，自己生成一条修复任务塞回队列 | 待做 |
| **规格拆解** | agent 读 spec 后调同一条 CLI | 把 maglev 等工具产出的清单变成任务 | 待做 |

已经落地的部分：任务数据模型与本地存储 · 侧栏入口 + 任务总览页 · 从任务直接开终端并把首包 prompt 带进 agent · 单次定时任务（到点自动开跑）· **完成边沿自动取下一条**（同目录下的待办排队串行跑）。

**待做：给 agent 一条写任务队列的 CLI**
后两种来源共用这一条接口——agent 干完一件事能自己往队列里塞下一件（完成即 drain 下一条，形成「人给一个起点，agent 自己往下推」的闭环），读完一份规格也能把它拆成一批任务塞进来（见下面第 2 点）。这是这一块的主要工作量，也是唯一要新建的对外接口。

> 边界：这里的循环是**事件驱动的自循环**（做完 → 取下一条 / 塞新任务），不是 cron 式定时重复跑；也不做完整项目管理和云端任务库——那不是驾驶舱该干的事。

### 2. 接 spec 工具的任务清单

spec 驱动的产物链是「意图 → 规格 → tasks 清单」，而清单是**死的**——谁在跑、跑到哪、卡住没有，文档自己不知道。smelt 手里恰好有那半截：活着的会话 + 状态感知。规格管「要做什么」，这边管「正在怎么做」。

**不写 spec 解析器，让 agent 去读。** 各家规格的组织方式都不一样，写 parser 就要维护一堆适配器，人家改个结构就得跟着改；而清单里的依赖关系、哪些能并行，正则也解析不可靠。读文档本来就是 agent 最擅长的事——何况 maglev 这类框架本身就是以 skills 形式跑在 agent 里的。所以由 agent 读完规格，调上面那条 CLI 建任务，顺带把依赖关系、能否并行、以及每条任务对应的那段规格一并写进去。

**所以这一路和「待做一」共用同一套机制**：agent 写任务队列的 CLI，自循环用它，导入 spec 也用它，不用做两套。smelt 只需要保证那个写入接口足够稳定和表达力够用。

任务进了队列之后，剩下的是 smelt 的活：

- **一条 task 一个会话** —— 不是把整份 spec 丢给一个 agent 从头跑到尾（上下文互相污染，中间挂了得整体重来），而是每条独立开会话：上下文干净、失败只重跑那一条、反复失败的还能单独换个 provider 重试
- **并行 + worktree 隔离** —— 互不依赖的同时开跑，各自一个 worktree 互不打架；有依赖的串行
- **带着规格片段跑** —— 每条任务自带它那段规格与验收标准，不用塞整份 spec（噪音大、烧 context），也不至于什么都不带（agent 会忘了要求）
- **状态回到清单** —— 跑完打勾、卡在等审批就标出来，而不是让 spec 文档假装一切正常
- **验收时 diff 与规格并排** —— 不用回头翻文档去记「这条当初要求什么」；不满意就行内评论喂回那个会话（见下面第 4 点）
- **把 agent 的提问沉淀回 spec** —— 执行时发现规格有歧义，agent 会停下来问人，而这个问答现在只存在于终端里，关掉窗口就没了。**能让 agent 卡住的地方，正是 spec 没写清的地方**——smelt 知道它什么时候在提问，也留着完整会话记录，能把「问题 + 答案」回灌给规格

不定义自己的 spec 格式、不做 spec 编辑器——这样才能同时兼容 maglev 和别家。

**配合 maglev：让它跑得更顺**

maglev 是一套以 agent skills + 协议落地的规格驱动框架，也就是说它**跑在 agent 里，也就跑在终端里**——smelt 是它的宿主，不是它的对接方。它的流程越严谨，「agent 停下来等人」的时刻就越多，而那正是 smelt 接住的东西。几个具体的着力点：

- **Atomizer 拆完就能并行开跑** —— 一份大 PRD 雾化成一批原子 issue 之后，现在得靠人一个个开终端、一条条喂过去。接上任务队列就是：拆完直接进队列，互不依赖的并行开会话，各自 worktree 隔离
- **Reflect 这一拍不再把流水线卡死** —— 三拍子里 Reflect 是强制的，agent 会停下来等人确认方案。人一旦不在电脑前，整条流水线就停摆几小时。有了状态感知 + 远程，手机上就能收到「哪一条在等你拍板」，点一下继续
- **十条并行时，知道该管哪一条** —— 拆出十个 issue 同时跑，有的卡在 Reflect、有的卡在权限确认、有的已经干完等验收。现在只能挨个切窗口看，smelt 这边是一屏扫完
- **interactive review 的结论直接回流** —— maglev 有交互式 spec / code review 的环节，评审出的问题现在要人从文档搬回终端。配合交互式 diff，就是在改动上写完评论直接喂回那个会话
- **长流程扛得住中断** —— 一个 feature 从 intake 走到 active 再归档是个长流程，中途关电脑、装个新版本，agent 全死。smeltd 让会话脱离 GUI 存活，这条流程才经得起真实的一天

> 这几条是读 maglev 的协议文档（`specs_lifecycle` / `collaboration`）和 skills 清单推断出来的，**没有实际跑过它的完整流程**——真实痛点还需要用过的人校正。

### 3. 动态注入上下文

agent 干得好不好，一大半取决于它手里有什么上下文。而人往往知道它不知道的事——「这个 bug 跟那个文件有关」「上次那个改动碰过这块」——但现在要把这些喂进去，只能手工找、手工复制、手工粘贴。**上下文喂不对，agent 会在错误的方向上跑很久，而这段时间的算力和你的注意力都白花了。**

已有的是手工版：右键文件 → 把 `@路径` 提及写进终端输入框；框选代码 → 带文件名和代码块发过去。

待做的是让它动态起来：

- **按需一键** —— agent 卡住或反问时，把相关文件 / 当前 diff / 失败日志一键送进去，不用自己翻自己打
- **自动备料** —— 按会话当前状态准备候选上下文（测试挂了就把失败日志和相关文件备好，等你点一下）
- **跨会话搬运** —— 把一个会话的结论或完整上下文喂给另一个会话；配合 Remix（换 provider 或换分支重跑）时不必从头再讲一遍
- **排队注入** —— agent 正在跑的时候不能往终端塞字节，那段文本会被当成它当前那个问题的答案。必须排队等它空下来才注入，UI 要如实显示「已排队 N 条，等 agent 空下来发送」，不能假装发出去了

**为什么这件事适合在 smelt 做**：它同时握着三样东西——上下文源（文件树 / git diff / 改动热力图 / 历史会话）、注入通道（PTY）、以及**时机判断**（会话 phase）。少任何一样都做不成：只有编辑器没有会话，不知道什么时候能注入；只有终端没有项目视图，不知道该注入什么。

### 4. 团队任务协作

远程能力已到 L3，L4 是下一步：

| 层 | 能力 | 现状 |
|---|---|---|
| L1 远程查看 | 看到会话在输出什么 | 已上线 |
| L2 远程知情 | 在跑 / 等你 / 问什么 | 已上线 |
| L3 远程操控 | 键入、批准 / 拒绝 | 已上线 |
| L4 团队协作 | 多人多任务、认领、移交 | **待做** |

关键是把协作的单位从「会话」提到「任务」：一条待认领的任务、一个卡在等审批的 agent、一份等 review 的 diff——机制上是同一件事，都是**一个等人处理的条目**。所以不做两套 UI，做一个**认领池**：什么都掉进来，谁有空谁点「我来」。

具体形态（详见 `docs/collaboration.md`，该文档标题即注明「设计讨论，未动工」）：

- **观战席** —— 给同事一条只读链接自己看进度，把「进度同步」这个纯负担自动化掉
- **IM 卡片** —— 状态变化推飞书卡片，直接在 IM 里点允许 / 拒绝，不用先打开链接

**冷启动约束**：团队功能最大的敌人不是技术，是「必须先说服第二个人装上 smelt」。所以硬性条件是**接收方零安装**（浏览器打开即可）——这一条直接决定了先做观战席和 IM 卡片，把需要全员安装的排到后面。

**参照：[multica](https://github.com/multica-ai/multica)**（Go + Web 看板，把 issue 像派给同事一样派给 agent）

**关系一句话：同一道题，站在权限门两边的两个答案。** 题目都是「一个人怎么指挥一群 agent」，分岔只在人要不要留在执行现场——multica 把人撤走，smelt 把人留下：

| | multica | smelt |
|---|---|---|
| **定位** | 派单平台：活派出去，回来看结果 | 驾驶舱：现场可见，随时介入 |
| **赌注** | agent 能自主跑完 | 人还得在场 |
| 执行 | 无头 spawn CLI，隔离工作区 | 真 PTY，完整终端 |
| 权限 | 绕过（`--yolo`），不阻塞 | 停下来等人批准 |
| 你看到的 | 消息流（结果摘要） | 实时画面（TUI 也能看） |
| 中途介入 | 报 blocker 后回应 | 随时打字进去 |
| 代价 | 安全边界让渡出去；卡住时只知道卡了，看不见卡在哪个界面 | 人的注意力是瓶颈，同时盯不了太多 |
| 适合的活 | 低风险、有把握：补测试、改文档、批量重构 | 高风险、要判断：改生产配置、动数据库、大重构 |

「无头 + 绕审批」不是推测：其文档对 Qoder 的启动命令是 `qodercli --yolo --acp`，明写「bypass permissions 模式，使工具调用在无头运行时不会阻塞在交互审批上」；全文没有任何 PTY / TTY 内容，回传的是按序号增量拉的结构化消息流，不是终端画面。

所以两者不是竞品二选一，而是同一条工作流里**按风险分档**的两段。

值得吸收的几点：

- **路由到组，而不是到具体的人** —— Squads：任务扔给一个组，由 leader agent 决定谁接。这和认领池是同一件事的推 / 拉两面，可以并存：有明确归属的直接派，没有的进池子等认领
- **agent 当一等成员** —— 有独立 profile、能发评论、能主动报阻塞、出现在活动时间线。对应到认领池就是：条目上要显示「谁在处理」，人和 agent 同格
- **任务生命周期由 agent 自主更新** —— enqueue / claim / start / complete / fail 全程自己推进并实时推送。smelt 的状态通道已经有 phase 感知，但任务侧还缺这套 claim / complete 语义
- **主动报阻塞** —— smelt 现在是从终端状态**推断**「它在等审批」，multica 是 agent **自己上报**。两者可叠加：上报更准，推断兜底（不改造 agent 也能用）

不吸收的：服务端 + PostgreSQL 部署（零服务端是 smelt 的冷启动优势），以及 Cron 式定时重复（见上面任务系统的边界说明）。


### 5. 接 ACP：无头执行的标准接口 + 走出终端的新皮肤

[ACP（Agent Client Protocol）](https://agentclientprotocol.com)是 Zed 发起的 agent ↔ 客户端标准协议：客户端 spawn agent 子进程，JSON-RPC over stdio 交互。它已长成行业标准——JetBrains 全线 IDE 与微软 Terminal 已接入，Zed 与 JetBrains 共推了 Agent Registry；Claude Code（官方适配器 `claude-agent-acp`）、Gemini CLI（原生）、Codex CLI、Copilot CLI（预览）、Qoder 都能跑，multica 的无头模式用的正是它。

对 smelt 来说这是**第二种会话类型**：PTY 会话是第一通道（完整 TUI、现场可见），ACP 是第二通道（结构化消息流）。同一批 agent、同一个驾驶舱，两种呈现，按风险和用户各取所需。价值按分量排：

- **原生界面，把用户群扩出终端** —— ACP 回传的不是终端画面，是类型化的消息流：对话、工具调用、diff、权限请求各是各的结构。smelt 可以给这种会话渲染原生 UI——对话流、工具调用折叠卡片、行内 diff、权限按钮（Zed 的 agent panel 就是这么做的）。终端是现有用户的舒适区，却是更大人群的门槛：习惯 IDE、不碰终端的人，从此也能用同一个驾驶舱指挥 agent。手机端受益更大——TUI 画面在小屏上天然难受（字号、宽度、重排都不是给手机设计的），消息流渲染成移动原生界面是体验质变，不再是「凑合能看」
- **远程审批换传输层** —— 现在手机批权限靠解析 TUI 菜单再注入按键，天然脆弱。ACP 的 `session/request_permission` 是带结构化选项（allow once / always / reject）的显式请求，手机页面和飞书卡片直接渲染成按钮，点了回一条 RPC。上一节 L4 的「IM 里点允许 / 拒绝」走画面解析很难做稳，走 ACP 几乎白送
- **状态从推断变协议事实** —— 四档状态（等批准 / 等输入 / 在跑 / 干完）在 ACP 里都是显式事件。这正是上一节「上报更准、推断兜底」的标准化形态，且不用改造任何 agent；OSC 推断保留，给非 ACP 会话兜底
- **无头执行通道** —— 「低风险活无头跑」不用发明 multica 那套私有消息流，spawn ACP 模式就是标准接口。附带红利：Remix 换 provider 变成换一条 spawn 命令，客户端代码不动
- **terminal 能力方向是反的，这是 smelt 的地利** —— 协议里 agent 跑命令是**请客户端代跑**（`terminal/create` 等）。smelt 做 ACP 客户端时，无头 agent 的每条命令依然跑在内嵌真 PTY 里——可见、可杀、可审，直接削弱上一节对比表里「无头卡住看不见卡在哪」的代价。multica 那种纯 Web 看板没有本地 PTY 地基，这块它做不了；smelt 的 PTY 积累在 ACP 时代不是包袱，是现成地基
- **排队注入的正解** —— ACP 的 prompt 是显式 turn，协议明说上一轮何时结束；注入就是排队发下一条结构化 prompt，文件 / diff / 日志作为 content block 带进去，不用伪装成键盘输入。上面第 3 点的时机约束在这里天然解掉
- **smeltd 顺势扩容** —— ACP agent 也是子进程，同样需要脱离 GUI 存活。smeltd 从「托管 PTY」扩展为「也托管 ACP 子进程」，重放 = 消息历史，比字节流重放还干净

两条边界：**ACP 与 TUI 是二选一的运行模式**——ACP 模式下 stdio 被 JSON-RPC 占用，agent 不渲染 TUI，所以这是新增一种会话类型，不是给现有 PTY 会话加持，驾驶舱的 PTY 通道照旧是高风险档的主场（也正因为拿不到 TUI，才要给 ACP 会话配原生皮肤）。**权限语义要逐家实测**——Copilot CLI 的 ACP 模式曾直接自动批准而不发 `request_permission`，协议里有权限门不等于每家都老实发；对赌「人还得在场」的 smelt，这是接入验收的第一条测试用例（另外 terminal 能力在 spec 里仍标 unstable，生态还年轻，OSC 推断这条腿不能砍）。


### 6. Git 增强：人类最重要的把控环节

**代码越来越多由 agent 写，人的把控点就越往后移——最后落在 diff 上。** 你不再逐行敲代码，但必须看清它改了什么、拦住不该合的东西。所以在 agent 时代，git 这一环不是变轻了，是变成了主控制面。

现在的 git diff 视图只能**看**（字符级行内高亮），把控链路到这儿就断了：看出问题得切出去、复制粘贴、再回终端喂给 agent——正是痛点 6 里「靠飞书拷贝」的那条路。

待做：

- **交互式 diff** —— 在 diff 上直接框选改动、写行内评论，批量打包发回 agent 会话继续处理，把「审查」和「指挥」接成一条链。这里有个只有执行层才碰得到的约束：agent 正在跑的时候不能往终端里塞字节（那段文本会被当成它当前那个问题的答案），评论必须**排队等它空下来**才能注入
- **Git worktree 集成** —— 在 UI 里建 / 删 worktree、在新分支上直接开会话，多个 agent 并行改同一个仓库互不打架
- **Remix** —— 把当前会话原样搬到隔离 worktree 继续跑，可切换 provider（Claude ↔ Codex），原会话的完整上下文带过去。同一个任务想换个模型或换条路试错，不用从头再讲一遍
